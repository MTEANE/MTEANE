03Day
Auth + Events Route
Dev A — API / Infra
POST /auth/register
Creates org, generates a 32-byte random key via crypto.randomBytes, stores its hash, and returns { org_id, api_key } once with a "store this — never shown again" warning.
src/middleware/auth.ts
Fastify preHandler: reads x-api-key header → hashes it → calls findActiveKeyByHash → attaches request.org. Returns 401 for missing key, 403 for invalid or inactive.
TypeScript type extension
Extend Fastify's request type so request.org is fully typed everywhere — no casting needed in route handlers.
Dev B — Worker / Engine
POST /events — route + schema
Fastify JSON schema: event_type required (pattern ^[a-z]+\.[a-z_]+$), payload required (any object), idempotency_key optional string.
src/db/queries/events.ts
Write insertEvent(orgId, eventType, payload, idempotencyKey?). The route calls this and returns { event_id, received_at, status: "received" }. No queue yet — just stored.
End of Day Can register an org, receive a key, and hit POST /events with it. Event is stored in DB.
04Day
Idempotency + Health
Parallel
Dev A — API / Infra
GET /health
Checks three things: SELECT 1 on Postgres, PING on Redis, process memory under threshold. Returns { status, db, redis, uptime } with appropriate HTTP code.
Auth middleware edge cases
Manually verify with curl: missing header → 401, wrong key → 403, inactive key → 403 with specific message, valid key → request proceeds. Document results in tests/auth.md.
Dev B — Worker / Engine
Idempotency logic in insertEvent
Before inserting, check if a row exists with WHERE org_id = $1 AND idempotency_key = $2. If yes, return the existing event_id — no new insert. Caller always gets 200 either way.
Unique DB index
Add UNIQUE INDEX idx_events_idempotency ON events(org_id, idempotency_key) WHERE idempotency_key IS NOT NULL. This is the safety net for race conditions — two simultaneous requests can't both insert.
End of Day Duplicate events with the same idempotency key are silently deduplicated. Health endpoint reports live status.

Blocked

Migration 003 — rules table
Columns: id, org_id (FK), name, event_type, condition (jsonb), action_type, action_config (jsonb), is_active, timestamps.
Partial index + getActiveRules

Dev B — Worker / Engine

09Day
Rule Evaluator
Sync First
Dev A — API / Infra
src/engine/evaluator.ts — core
Main export: evaluate(condition, payload): boolean. The field supports dot notation — "order.amount" resolves to payload.order.amount via a resolvePath(obj, path) helper.
Operators: eq, neq, gt, gte, lt, lte
Numeric comparisons cast both sides to Number before comparing — so "100" and 100 both work with gt.
Dev B — Worker / Engine
Operators: contains, startsWith, endsWith, exists, in, regex
contains works on strings and arrays. exists checks field is present and not null. regex tests field against a pattern string. All added to the same evaluator file.
src/engine/evaluator.test.ts
Minimum 15 test cases. Must cover: nested dot paths, null values, type mismatches on numeric ops, missing fields with exists, array in checks, and malformed conditions that should return false not throw.
End of Day All operators work correctly. 15+ tests passing. Evaluator never throws — returns false on bad input.
10Day
Evaluator → Worker
Blocked
Dev A — API / Infra
Matching loop in worker
For each rule from cache, call evaluate(rule.condition, event.payload). Collect matched rules. If a rule's condition is malformed and throws, catch per-rule and continue — one bad rule should not fail the entire job.
Log match results
Log { matched: N, total: M } per job. Zero matches is normal, not an error — log it as info, not warn.
Dev B — Worker / Engine
Migration 004 — action_logs
Columns: id, event_id (FK), rule_id (FK), org_id, status (pending | success | failed | retrying | dead), attempt_count, error_message, response_body, executed_at.
Insert pending action_logs
After the matching loop, insert one action_log row per matched rule with status: 'pending'. These are the paper trail — if the process dies before executing, you know what was supposed to happen.
End of Day Worker evaluates all rules, identifies matches, and creates pending action_log entries for each.
✓ Phase 3 Complete — Rule engine works end-to-end. Events are matched against conditions. Matches are logged.

14Day
Dead Letter Queue
Blocked
Dev A — API / Infra
src/queue/dlqQueue.ts
Separate BullMQ Queue named "dead-letter". Listen for the worker's "failed" event — fires after all retries exhausted. Move job to DLQ with original data plus { failedReason, finalAttempt }.
GET /dlq
Calls dlqQueue.getFailed() and returns jobs filtered to the authenticated org. Gives orgs visibility into what failed and why.
Dev B — Worker / Engine
POST /dlq/:job_id/retry
Fetches job from dead-letter queue, validates it belongs to the requesting org, re-adds it to the main "events" queue with fresh retry counters. Updates related action_log rows back to status: 'retrying'.
Mark action_logs as dead
When a job exhausts all retries, update its action_log rows to status: 'dead'. This is the DB-side mirror of the queue-side DLQ — both stay in sync.
End of Day Failed jobs land in DLQ. Orgs can list them and manually retry. DB logs reflect dead status.
15Day
Rate Limiting
Parallel
Dev A — API / Infra
src/middleware/rateLimiter.ts
Redis sliding window via Lua script: ZADD current timestamp → ZREMRANGEBYSCORE entries older than 60s → ZCARD count remaining. All three commands in one atomic Lua script to prevent race conditions.
429 with Retry-After
When limit exceeded, return 429 with a Retry-After header set to the seconds until the oldest entry in the window expires.
Dev B — Worker / Engine
src/config/plans.ts
Map plan names to limits: { free: 100, pro: 1000, enterprise: 10000 } events/minute. Middleware reads request.org.plan (already attached by auth) to pick the correct limit.
Rule count limit on POST /rules
Before inserting a new rule, count existing active rules for the org. If count >= planLimit.maxRules, return 403 with message "Rule limit reached for your plan". Free plan cap: 10 rules.
End of Day Per-tenant rate limiting active on POST /events. Plan-based rule caps enforced on rule creation.
16Day
Resilience Hardening
Parallel
Dev A — API / Infra
Cache invalidation completeness
In PATCH /rules/:id, if event_type is being changed, invalidate cache for both the old and new type. A stale cache for the old type would keep firing rules that no longer apply.
DB timeouts + latency monitoring
Set statement_timeout: 5000 on the connection pool. Add a 30-second interval in db/healthCheck.ts that logs a warning if a test query exceeds 200ms.
Dev B — Worker / Engine
Duplicate job guard
Before queue.add(), call queue.getJob(eventId). If the job exists and is waiting or active, skip enqueuing and log a warning. Prevents double-processing when clients retry before the first job completes.
Configurable worker concurrency
Read concurrency from WORKER_CONCURRENCY env variable, default 5. Document it in .env.example with a note on when to tune it (CPU-bound vs I/O-bound workloads).
End of Day System handles failures, abuse, and restarts without data loss or duplicate processing.
